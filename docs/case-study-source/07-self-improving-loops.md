# A design system that grades its own homework

*Case-study source draft — narrative voice, for `docs/system-case-study.html`. Not yet linked from the docsify sidebar.*

## What twelve runs of receipts already say

`run-ledger.json`, the append-only record of every full `/review-component` run, has accumulated enough entries to answer a routing question with data instead of a guess: the adversarial reviewer caught real issues beyond what the deterministic gate could catch on every interactive or composite component it saw — Accordion, ButtonArrow, DropdownMenu, TextField, Chip, TextLink — and found nothing beyond the gate on any layout primitive: Box, Stack, Inline, Heading, Checkbox, all zero. A 100% catch rate on one class of component and 0% on another isn't a hunch, it's a pattern the ledger surfaced on its own — spend the fresh subagent where it earns its cost, and let primitives take the cheaper path.

That's the shape of every mechanism in this chapter: this system has four different loops that make some part of itself demonstrably better over time, and the honest throughline across all of them is smaller than "self-improving" implies. It's closer to a system that grades its own homework — checks its own outputs against evidence, and only changes its own rules when the evidence says to.

## Docs that police their own readability

The docs site serves three audiences who read it very differently — a product manager, a product designer, and a software engineer — but for a long time it was written with only the last one in mind. `docs/08-glossary.md` was the exception: written in plain language on purpose, for the other two. An audit found the rest of the pages failing a fixable, repeating pattern — sentences stacking three new concepts at once, technical terms used without a link to the glossary entry that already existed for them, no signal for which sections assumed engineer vocabulary and which didn't.

The fix follows the same detection-is-a-script, judgment-is-triggered split used everywhere else in this system: `npm run docs:check` detects when a page has gone stale relative to the source files it depends on, and the developer-triggered `/docs-sync` moment does the actual rewriting. What's new is what happens between the rewrite and the pull request — a **scribe**. A fresh, read-only subagent with no memory of the rewrite session reads only the sections that were just rewritten and judges them purely on first-read comprehension, checking each technical term against a closed, three-part test: is it defined in plain language on first use, is it linked to its glossary entry, and can the section's declared audience be reasonably assumed to already know it. The finding types are a closed set — seven of them, no room to invent an eighth — and the prime directive is "define, don't delete": a technical term is never a bug to be removed, only a term that needs grounding.

That rubric wasn't designed clean in one sitting. It was corrected by a real failure. While hand-fixing a "Glob pattern" glossary entry, the reviewer signed off on a definition that itself leaned on an undefined word — "wildcard" — and after that got patched, a second undefined word turned up one level down: "file path." The original rubric checked the headword being defined but had no visibility into the words the definition depended on. The fix is a recursion rule, one level deep only: whenever the scribe reviews or proposes a definition, it also runs the same three-part test against every technical term used *inside* that definition — but it stops there, flagging anything deeper as a separate finding for a human to decide how far to keep unwinding, rather than chasing the chain to the bottom automatically. It's a small rule, but it's the kind of rule you can only write after watching the naive version fail in a specific, recorded way.

## The honest-outcome harness

The system's clearest demonstration of "measure, don't assert" is what happened to an idea that sounded obviously good. Once a cross-component pattern file existed — a scan of all the components grouped by shared interaction pattern, with drift flagged between what a component's code does and what its own metadata claims — the natural next move was to feed it into every generation task the system runs. More structural context, the reasoning went, can only help.

Instead of shipping that assumption, the system tested it. Seven tasks were pre-registered before any results came in — locked in advance, so nobody could quietly redefine success after seeing which arm won. Each task ran twice: once with only per-component metadata in the prompt, once with the pattern file added on top. Both arms were scored against the same real deterministic gates the system already runs in CI, plus a trap checklist looking for specific known mistakes. The result wasn't the clean win the intuition predicted. Composition and layout tasks improved substantially — violations roughly halved, one task reaching zero. Component scaffold tasks got measurably worse. The likely mechanism, written down rather than left as a shrug: a large aggregate file crowds out the model's attention on the narrower schema and file-contract rules a scaffold task actually needs to hit.

The system shipped only the half that measured well. The pattern file went into `/layout-generation`'s prompt and nowhere near `/component-scaffold`'s — a scope decision a harness produced, not a hunch. And then the harness did something a lot of one-off experiments don't: it stayed in the repository, unused day to day, specifically so the same question could be re-asked with real numbers if the inputs ever changed again. It didn't have to wait long to earn that shelf space. A later pass found a real bug in its own scoring — a false positive that fired whenever a text prop was passed to a component in attribute position, which is actually correct usage, not a violation. The fix exempted that position, and every one of the 14 scored cells was recalculated from the retained raw outputs, no tasks rerun. The corrected numbers didn't flip the conclusion; they sharpened it. That's the detail worth sitting with: the instrument caught its own mistake, the correction is dated and on the record as an ADR amendment, and the decision it existed to inform didn't have to be revisited from scratch.

## Keeping receipts in case the system fires its own reviewer

Every `/review-component` run spawns exactly one subagent — an adversarial reviewer, deliberately skeptical, looking for reasons the code is wrong rather than reasons it's fine. That subagent costs something real: it's a usage-window expense on a Claude Pro plan, where the constraint isn't dollars per token but a shared weekly allowance. Whether that cost is worth paying is a fair question to ask, and rather than answer it with a shrug, the system keeps a receipt every time.

`run-ledger.json` is append-only and committed — nothing in it is ever edited or deleted, only added to. Each entry records what a `/review-component` run actually found, and the single most important field in it is a count of findings the reviewer caught that the deterministic gate — type checking, build, metadata validation, accessibility coverage — could not have caught on its own. The ledger is now old enough to read: across its 12 full-loop runs, the gate passed everything it could check, the reviewer added findings beyond the gate in 7 of 12 runs (18 findings, 1.5 per run on average), and not one run needed a manual rescue. The routing heuristic in the intro above is exactly the kind of decision this file exists to make possible: made from receipts, revisable when the receipts change ([The numbers](08-measured-impact.md)). **Measured** claim, tagged plainly: this is 12 runs, 12 of 27 shipped components (44%) — the cheaper in-session path records no findings telemetry at all, so the routing heuristic is real for the full-review path and unmeasured for the rest.

The same discipline was written into the newer docs-scribe critic before it even had a track record — **instrumented, not yet measured**: the ADR that created it commits, up front, to judging it by the identical ledger-and-routing method once it has accumulated a few real runs, but as of this writing it hasn't run enough to have its own numbers. The system isn't just willing to grade its reviewer's homework — it's set up so that grade could, one day, be the reason the reviewer's role changes.

## Ledgers designed to die

Two lists in this system exist purely to shrink. One tracks components that predate the rule requiring a behavioral accessibility test and were waived in the meantime while the backlog got worked through. The other tracks known color-contrast failures, each one carrying a reason and a linked issue rather than sitting silently unexplained. Both lists came with a rule stronger than "please shrink these eventually": new entries are structurally forbidden. A newly built interactive component cannot be added to the accessibility waiver list at all — it has to ship its test. A newly discovered contrast failure doesn't get waived; it gets routed into a curated, reviewed list of color pairings instead. The accessibility backlog has already reached the state it was designed to reach: empty. That's not a metaphor for progress — it's a literal, checked-in `[]` in a JSON file, proof that a ledger built to die can actually die instead of just getting smaller in someone's head.

## Gates that check, and loops that learn

It would overstate the case to call everything automatic in this system a "learning loop." Plenty of it is a pure gate: a fixed check that passes or fails against the current state and does nothing else — validating a component's metadata against its schema, confirming a layout file follows the landmark grammar, detecting that a documentation page has drifted from the source files it depends on. Those checks are essential, and none of them change what they check for next time; they're backstops, not memory. What earns the "self-improving" label in this system is narrower and more specific: a finding that gets routed into a metadata file so the next agent inherits it as a checked fact instead of a story someone has to remember to retell; a ledger that accumulates real outcomes so a costly step can be judged on evidence instead of defended by conviction; a waiver list built with a one-way door so debt can only ever shrink; and a handful of scripts that keep the system's own instructions and its own documentation honest without anyone having to remember to check. Four different shapes, one shared discipline: don't assert that something works. Write down what happened, and let that answer the question later.
